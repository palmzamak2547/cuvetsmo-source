# Contributing to source.cuvetsmo.com

> Iron Rule 0: no fabrication. Every dose, indication, contraindication
> that ships canonical must have a verifiable citation chain. AI may
> draft, never authors. Humans are accountable for every line.

This document is the contract. PRs that violate it bounce.

---

## What this repo is

A verified Thai medical knowledge layer. Every entry has:

1. **Citation chain** linking back to peer-reviewed source, regulatory database, or textbook
2. **Mirror provenance** showing the exact authoritative source the structured data came from
3. **Faculty signoff** with named reviewer, title, department, affiliation, date
4. **Cryptographic signature** by the reviewer's Ed25519 key (verifiable in the browser by any reader)
5. **Ontology codes** (ATC, RxNorm, and where licensed SNOMED CT, LOINC, ICD-11) so downstream systems can join on stable IDs
6. **Drafting metadata** declaring honestly whether AI drafted any of the text and how much a human edited

If your PR cannot satisfy all six, do not open it — open an Issue first to discuss scope.

---

## Iron Rule 0 (the only one that matters)

**No claim ships canonical without a verifiable chain back to an authoritative source.**

Concrete consequences:

- A faculty reviewer who has not personally read the cited source must not sign off
- A draft generated by AI that a human "approved without changes" cannot ship as canonical
- A dose copy-pasted from another site, ChatGPT, or a study guide cannot ship — only from the original authoritative document (DailyMed, WHO EML, peer-reviewed paper, textbook with page number, regulatory database)
- A translation that paraphrases beyond the source intent must be flagged and edited back
- "Template" placeholder text must be stripped before `reviewedBy` is set

If you find a claim on this site that does not satisfy Iron Rule 0, file a GitHub Issue with the label `iron-rule-0-violation`. Volunteer or faculty maintainers treat those as P0.

## AI in the loop, never as the authority

AI is a tool inside the editorial pipeline. It is **never** the named source of a canonical claim.

What AI is allowed to do:
- Draft Thai translations of English source material
- Summarize long monographs for the reviewer
- Suggest ontology codes that the reviewer verifies
- Rank candidate citations for relevance
- Lint Thai grammar / clinical-terminology consistency

What AI is **never** allowed to do:
- Be cited as a source (no `type: "ai-output"` citation type — does not exist)
- Have its output ship canonical without `humanEditsRatio >= 0.1` and a human reviewer
- Sign on behalf of a faculty member
- Generate dose numbers, indications, or contraindications without a corresponding citation to an authoritative source

When AI is used:

```json
"drafting": {
  "aiAssisted": true,
  "aiModel": "claude-sonnet-4.7",
  "aiPromptHash": "<SHA-256 of the prompt>",
  "humanReviewer": "ekkapol.akb",
  "humanReviewedAt": "2026-06-15",
  "humanEditsRatio": 0.62
}
```

The `humanEditsRatio` field is calculated as approximate byte-diff between the AI draft and the final committed text. Less than 0.10 means the human did not meaningfully edit — the lint blocks it. We trust the contributor to compute it honestly. The faculty reviewer cross-checks during signoff.

For pure-human entries:

```json
"drafting": {
  "aiAssisted": false,
  "aiModel": null,
  "aiPromptHash": null,
  "humanReviewer": "ekkapol.akb",
  "humanReviewedAt": "2026-06-15",
  "humanEditsRatio": null
}
```

## How content lives on disk

```
content/
  drugs/<slug>.json              Canonical drug entry (one file per entry)
  citations/<cid>.json           Content-addressed citation metadata
  keys/<kid>.pub.json            Signer public keys (committed)
  keys/index.json                Key registry
  log/transparency-log.jsonl     Append-only signed log of all signatures
  signatures/                    Future: detached .sig files (Week 5+)
```

Everything is a flat JSON file. Git history is the audit trail. PRs are the editorial workflow.

## Contribution flow (new drug entry)

1. Pick a drug. Search existing entries first to avoid duplicates.
2. Open an Issue first if scope is non-trivial (≥ 5 dosages, complex contraindication tree, or species-specific edge cases).
3. Fork the repo, create a branch `entry/<slug>`.
4. Scaffold from the schema (`lib/drugs.ts` is the source of truth for required fields).
5. Mirror an authoritative source. For NSAIDs / antibiotics / common drugs, DailyMed is the first stop. For Thailand-specific availability, check Thai FDA. For WHO classification, check the EML.
6. Compute mirror CIDs. Until `scripts/mirror.ts` ships in Week 2+, stub CIDs use the metadata-only protocol documented in `lib/cid.ts`.
7. Add ATC and RxNorm codes at minimum. SNOMED CT codes are accepted but not verified locally (licensing pending).
8. Mark `drafting.humanReviewer` with your GitHub username. Set `drafting.aiAssisted` honestly.
9. Run `npm run check` locally. Fix any lint errors before opening the PR.
10. Open the PR, fill the template, request a faculty reviewer.

## Faculty review flow

Faculty reviewers are the named humans who carry clinical accountability.

1. A maintainer assigns you a PR (or you self-assign).
2. Read every clinical section. Strip TEMPLATE markers, fix Thai phrasing, adjust for Thai veterinary context.
3. Cross-check at least one cited dose against an independent authoritative source.
4. Add your reviewer record:
   ```json
   "reviewedBy": {
     "name": "ผศ.น.สพ.ดร. นามจริง",
     "title": "ผู้ช่วยศาสตราจารย์",
     "department": "ภาควิชาเภสัชวิทยา",
     "affiliation": "คณะสัตวแพทยศาสตร์ จุฬาลงกรณ์มหาวิทยาลัย",
     "date": "2026-06-15",
     "did": "did:web:source.cuvetsmo.com:faculty:nam-jing-cuvet",
     "signerKeyId": "ed25519:<fingerprint>"
   }
   ```
5. Run `npm run check`.
6. Maintainer (or you, if you hold the key) signs the entry:
   ```
   node scripts/sign.mjs <slug> --signer <your-kid>
   ```
7. Maintainer merges. The entry flips from amber pending to emerald canonical.

## Citation rules

- **Every clinical sentence must carry at least one `cites: ["<id>"]` reference.**
- Citation `id` must resolve in the entry's `citations[]` array. Lint blocks dangling refs.
- Citation `url` must point at the document being cited. No paywall workarounds, no archive-only links unless the original is dead.
- For book citations, `pageOrSection` is required.
- For paper citations, year is required.
- For regulatory-database citations, `cid` is encouraged (so when the upstream changes, our snapshot is the canonical reference).

A citation that does not exist in our `citations[]` cannot be referenced. Add it first, then cite it.

## Mirror provenance

If your entry's structured content was derived from a mirror source (most entries are), declare it in `mirroredFrom[]`:

```json
"mirroredFrom": [
  {
    "id": "dailymed",
    "name": "NLM DailyMed",
    "url": "https://dailymed.nlm.nih.gov/",
    "entryUrl": "https://dailymed.nlm.nih.gov/dailymed/search.cfm?query=meloxicam",
    "asOfDate": "2026-05-27",
    "license": "U.S. Government work — public domain. Attribution recommended."
  }
]
```

Each entry in `mirroredFrom` must have a matching `mirrorCIDs` record. Lint enforces it.

## Signing

Faculty signing requires a registered key. To register a new signer:

1. Generate a keypair locally:
   ```
   node scripts/keygen.mjs <kid> --display "Display Name"
   ```
   The public key lands at `content/keys/<kid>.pub.json` (committed). The private key lands at `~/.cuvetsmo-keys/<kid>.priv.json` (NEVER committed — outside repo by design).

2. Commit the public key in a separate PR titled `keys: register <kid>`. Include in the PR description the signer's name, affiliation, and (if applicable) the faculty unit they review for.

3. Once merged, `scripts/sign.mjs <slug> --signer <kid>` works from the signer's machine.

**Private key safety:**
- Never email, paste, screenshot, or store in cloud
- For production faculty, the Phase 0 software-key model will be replaced with hardware-backed signing (YubiKey or equivalent) by Phase 2
- If a private key is compromised, file an Issue with label `key-rotation`. The compromised public key gets revoked, a new key gets registered, and all previously-signed entries by that key are re-signed

## Schema changes

Schema lives at `lib/drugs.ts`. Any change to the `Drug` type requires:

- Migration plan documented in the PR description (how existing entries are updated)
- `scripts/verify.mjs` updated to enforce new fields
- `.github/PULL_REQUEST_TEMPLATE.md` updated if checklist items change
- ARCHITECTURE.md updated if a primitive's surface changes

Schema changes that break existing entries (e.g., removing a field) require all existing entries to be migrated in the same PR. Lint blocks half-migrated states.

## What this project is NOT

We do not:
- Provide clinical advice — readers must consult a licensed veterinarian
- Replace MIMS, Plumb's, or UpToDate — we are a complement, not a substitute
- Accept paid content placements — no advertising, no sponsored entries
- Generate content without human review
- Distribute pirated textbook content — only open-license sources

If your contribution would push this site toward any of the above, it gets declined.

---

## Code of conduct

- Discuss in Thai or English, both welcome
- Cite sources, not opinions
- Disagreement on clinical detail = open a discussion thread on the PR, ping a second reviewer if needed
- Personal attacks, gendered language, or any form of harassment toward contributors or faculty: immediate ban
- We are building a long-term knowledge commons. Behave accordingly.
